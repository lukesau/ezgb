# Investigating the 1.05e instability reports

## The question

Was the instability people saw after updating to kernel 1.05e/FW5 a bug in the update itself,
or a consequence of users updating with slow SD cards / weak console batteries?

## What EZ Flash's own release notes establish (paraphrased from their distribution page)

EZ Flash distributed 1.05e/FW5 labeled explicitly as a **release candidate**, not a final
release, with their own warning attached: updating requires a high-quality, high-speed
microSD card, since a slow/low-quality card during the *update itself* can brick the
cartridge; they also required fresh or externally-powered batteries in the console during the
update. Separately, and after a successful update, they acknowledged a **known runtime
bug**: consoles with a slower SD card would hang at the LOADING or OS INIT screen post-upgrade.
They recommended staying on FW4/1.04e unless RTC game support or a specific GBC hardware
revision fix was needed.

The changelog's stated goals for this RC were: supporting CGB units without a CPU suffix
(tied to a "Micro SD Initial Error" issue on those units), a rewrite of the RTC handling so it
works correctly with official RTC games, and a faster ("turbo") loading speed on CGB and later
hardware.

**This settles the framing**: 1.05e was shipped as a Release Candidate, explicitly *not* a
final release, with an officially acknowledged known issue that matches what you'd expect from
the "slow SD card" hypothesis — but it's described as a **runtime** hang (LOADING/OS INIT,
after a successful update), separate from the **bricking-during-the-flash-process** warning
(which is about the update procedure itself, not the resulting kernel's behavior).

So there are two distinct risk windows, both real, both documented by EZ Flash:
1. **During the update**: slow card + weak console batteries → bricked cart (a flashing-
   process risk, not a kernel bug).
2. **After the update, ongoing**: slow card → hang at LOADING/OS INIT (an RC-quality runtime
   bug, acknowledged and left unfixed in this build).

Hash-verified: the `ezgb.dat` / `Update_FW5_7-31.gb` in `juniorkernel-1.05e-FW5/` match the
SHA1s EZ Flash posted (`ce1d531...`, `80ef33a...`) — confirmed with `shasum` against this
repo's copies, so we know we're analyzing the genuine RC build, not a corrupted/tampered copy.

## What the disassembly adds

From the 1.04e → 1.05e diff (see `re/REGISTERS.md` and bank-level diff work in the main
session): the real logic changes are isolated to bank 0 and bank 1. Several clean insertions
in bank 1 (not present in 1.04e) contain:

- A **NOR flash write sequence**: `ld bc,$4000 / ld a,$11 / ld [bc],a` (a standard MBC-style
  ROM-bank-select write) immediately followed by `ld bc,$a000 / ld a,$aa / ld [bc],a` — `$AA`
  is the classic first byte of a JEDEC NOR flash unlock/write command. This lines up with the
  changelog's "RTC codes are rewritten": the RTC value is very likely persisted to onboard NOR
  flash rather than a battery-backed RTC chip register. (`re/1.05e/disassembly/bank_001.asm`
  around line 4310-4320, function context starting at `Jump_001_5657`.)
- A `TIME:` / `SET` / `AUTO SAVE:` string block in bank 0 (offset `0x11910` in the ROM) — a SET-menu
  screen for viewing/setting the clock, confirming there's a user-visible, settable time value
  behind this RTC rewrite.
- The existing `BATTERY` / `DRY!!!` low-battery warning (from 1.04e, offset `0x1911` in the ROM)
  is unrelated to this — it's the console-battery warning already shipped in 1.04e's "Added a
  battery dry notice," not an RTC backup battery indicator.

No CGB-revision hardware registers (`$ff4c`-`$ff77` range) are touched anywhere in either
kernel version — if "CGB without CPU suffix" detection is real, it's happening through the
cart's own custom FPGA registers (`$7fxx`, see `re/REGISTERS.md`), not by reading GBC's own
documented/undocumented CPU registers directly.

## Open questions / not yet confirmed

- Whether "onboard battery" in your hypothesis means the *console's* AA batteries (matches
  the official warning directly) or an *RTC backup cell on the cart itself* (plausible given
  the NOR-flash RTC persistence found above, but not confirmed — would need to physically
  inspect a Jr PCB for a coin cell, or find NOR-flash RTC read/init code and see if it depends
  on a battery-backed component vs. just flash wear-leveling).
- Whether the "turbo loading speed" change (faster SD clock/timing for CGB+ devices) tightens
  timeout/retry assumptions in a way that would explain hangs specifically on slower cards —
  haven't yet located the SD wait-loop's timeout constant in 1.05e to compare against 1.04e's
  equivalent. This is the most direct way to confirm or rule out the "runtime hang" mechanism
  and would be the next static-analysis step if this thread gets picked back up.
- Task #4 (dynamic tracing in SameBoy) is still the fastest way to nail down the actual loader
  path and would likely resolve the open questions above faster than continued static digging.
